# 🏛️ 금고지기 (Vaulter) 마스터 플랜 및 기획서

- **업데이트 일시:** 2026년 4월 15일 12:41 (KST)

## 1. 제품 철학과 비전 (Philosophy)

- **Anti-가계부:** 유저를 통제하고 잔소리하며 죄책감을 주는 기존의 `가계부` 패러다임을 버린다.
- **Premium Vault:** 내 자산을 안전하게 보관하고 불려주는 스위스 은행의 `개인 금고`이자 `VIP 자산 관리실`을 지향한다.
- **Zero Entry (수기 입력 0%):** 유저는 영수증을 던지거나 메일 권한만 주면, AI가 모든 것을 분류한다.
- **100% Privacy (로컬 퍼스트 & BYOC):** 유저 데이터는 회사 서버에 단 1바이트도 저장되지 않는다. 철저히 기기 내(Local)와 유저 본인의 클라우드(Private Sync)에만 존재한다.

## 2. 핵심 UI/UX 아키텍처

### 2.1 대시보드 레이아웃 (지기)

- **좌측 패널 [나의 현실]:** 총자산, 순자산, 가용 현금 등 팩트 기반 데이터.
- **중앙 패널 [나의 꿈/목표]:** 가족 여행, 새 기기 구매 등 목표 진행률을 통한 성취감 고취.
- **우측 패널 [AI 비서]:** AI 챗봇과의 대화 및 동적 액션 칩을 통한 명령 하달.
- **하단 패널 [동적 데이터 원장]:** 상황에 따라 타이틀이 실시간으로 변하는(Dynamic Context) Seamless 타임라인 원장.

## 3. 메인 메뉴 (GNB) 구조

- **[거래내역] 메뉴 전격 폐지:** 메인 대시보드(`지기`) 하단 패널로 원장 완벽 통합.
- **지기 (Keeper):** AI 비서, 시각화 패널, 데이터 원장이 통합된 메인 통제실.
- **황금자산:** 나의 부(Wealth)가 축적되는 프리미엄 포트폴리오.
- **예산&목표:** 미래를 위한 계획과 지출/알림 세팅.
- **비밀금고:** 계약서, 보증서, 영수증 등을 프라이빗하게 보관하는 수장고.

## 4. 코어 엔진 탑재 마스터플랜 (The Engine)

- **무거운 로컬 AI 완전 폐기:** 부정확한 오픈소스 OCR(Tesseract/Paddle)과 무거운 로컬 LLM을 걷어내고, 압도적 성능 + 가벼운 프론트엔드를 위해 클라우드 API 하이브리드 채택.
- **The Brain & Eyes (GPT-4o-mini Vision):** 영수증/문서 업로드 시 일회성으로 OpenAI 비전 API를 호출해 완전한 JSON으로 구조화 후 원장(PENDING)에 삽입. (데이터 처리 즉시 서버 휘발)
- **The Storage (BYOC Private Sync):** 로컬 DB(IndexedDB)를 무조건적 연산/조회 기준으로 삼고, Google Drive(또는 iCloud)를 거울처럼 양방향 백업(Shadow Sync) 용도로 사용해 다중 기기 연동 구현.
- **The Tollgate (수익화):** API 호출 시 우측 상단 크레딧을 차감하는 통합 마이크로 과금 로직 적용.

## 5. 🚀 70-20-10 수집 파이프라인 (Zero-Entry Automation Vision)

수기 입력을 완전히 멸망시키고 가계부를 방치하는 90% 유저를 구원하기 위한 3단계 자동화 룰:

### [The 70%] Gmail 비서형 백그라운드 수집 (Core Automation)

- **대상:** 온라인 쇼핑, 넷플릭스 구독 등 매일 쏟아지는 이메일 결제 내역.
- **방식:** Gmail API Read-Only 연동을 통해 Service Worker가 백그라운드에서 조용히 메일을 수집하고, AI 파싱 후 로컬 DB에 적재. (Silent Fetching)
- **UX (Digest Notification):** 실시간 알림 남발 대신, 유저가 지정한 시간(예: 저녁 8시)에 `오늘 O건이 자동 정리되었습니다` 형태의 우아한 1회 요약 알림 제공.

### [The 20%] CSV/엑셀 벌크 업로드 (Monthly Batch)

- **대상:** 월 1~2회 밀린 카드사/은행 내역을 한 번에 정리하려는 유저.
- **방식:** 포맷과 무관하게 엑셀/CSV를 업로드하면, AI가 열(Column) 구조를 자동 파악하여 수백 건 장부를 일괄 파싱 및 카테고리 분류.

### [The 10%] 스마트폰 카메라 스캔 & 공유 (Edge Cases)

- **대상:** 현장 결제 종이 영수증, 계약서 등 소수의 아날로그 증빙.
- **방식 1:** 앱 내 카메라 업로드 → GPT Vision 파싱.
- **방식 2:** 카카오톡 알림톡 결제 내역 등을 PWA `Web Share Target`으로 스마트폰 공유창에서 Vaulter로 즉시 전송 및 파싱.

## 6. Zero-Friction Onboarding (점진적 연동 UX 전략)

초기 유저의 진입 장벽(구글 로그인 등)을 0%로 만들기 위한 온보딩 UX 설계:

- **Step 1 (가입 없는 체험):** 랜딩 즉시 로컬 DB만으로 앱 100% 구동. 영수증 업로드 성능 체험.
- **Step 2 (넛지 유도):** 3~4번째 사용 시 `모바일에서도 보시려면 드라이브에 안전하게 프라이빗 싱크 하세요` 넛지 제공.
- **Step 3 (디폴트 전환):** 유저 동의 후 OAuth 권한 부여 시, 로컬 데이터를 클라우드로 업로드하며 영구적 다중 기기 동기화 모드 진입.

## 1. 프로젝트 비전 및 핵심 철학

### 1.1 패러다임의 전환: "가계부를 죽여라"

| 현상의 문제 | 우리의 해답 |
|---|---|
| 기존 가계부는 '입력 노동'을 강요하며, 80% 이상이 중도 포기 | 기록(Recording)이 아닌 **보관(Archive)**과 **해석(Interpretation)**에 집중 |

**핵심 가치 (3 Core Values):**

1. **Zero Entry** — 수기 입력 제로. 파일 업로드와 로컬 스캔만으로 데이터 자동 생성.
2. **Absolute Trust (단일 기기 원칙)** — 유저의 민감 재무 데이터는 오직 로컬에만 저장. 서버로는 AI 분석에 필요한 최소한의 정보만 전달.
3. **Insight First** — 숫자의 나열이 아닌, 문장과 전략으로 돈의 흐름을 설명.

### 1.2 서비스 정의 및 보안 철학: "진짜 금고는 옮겨 다니지 않는다"

- **포지셔닝**: 흔한 금융 앱이 아닌, 집안의 모든 돈 관련 자료를 보관하고 분석하는 **'개인 재무 운영체제(Money OS)'** 및 **'재무 금고(Vault)'**.
- **아날로그적 보안 철학 (Self-Responsibility)**: 서비스 차원의 자동 클라우드 백업 미제공. *"백업이 필요하다면 유저 각자의 방식으로 다양하게 선택해 신중히 보관하라"*고 가이드하여 '금고'의 신비감과 소중함 강조.

                    │

### 4.2 Ingestion & Cleanup Engine (데이터 투입/정제)

- **멀티 소스 처리**: 은행 CSV/XLSX, 카드 명세서 PDF 등 드래그 앤 드롭 업로드
- **OCR 이원화 전략**:
  - **기본 (로컬)**: 한국어 특화 오픈소스 OCR (WASM). 최소 과금 (광고 대체).
    - User-in-the-loop: 내용/거래처/금액 컨펌 과정에서 유저가 간단히 수정
  - **프리미엄 (서버)**: 고정밀 유료 OCR API. 크레딧 차감.
  - *"당신의 영수증은 기본적으로 이 브라우저 안에서만 읽힙니다. 더 정확한 인식을 원하시면 프리미엄 스캔을 선택하세요."*
- **내부 이체 식별 & 중복 제거**: [일시]+[금액] 대조 → 계정 간 이동 및 이중 계상 방지

### 4.3 Insight Engine (AI 분석/해석) — 하이브리드 AI 구조

| 레이어 | 기술 | 역할 | 과금 |
|---|---|---|---|
| 로컬 AI | WebLLM (WebGPU) | 단순 분류, 파싱, 카테고리 추천 | 최소 과금 (광고 대체) |
| 서버 AI | Gemini 2.5 Flash (서비스 API Key) | 고도 추론, 미래 시뮬레이션, 복잡한 재무 상담 | 크레딧 차감 |

- **User-in-the-loop**: 애매한 분류 시 AI가 신뢰도를 억지로 매기지 않고 유저에게 대화로 질문. 유저가 최종 컨펌(승인)한 데이터만 저장.

### 4.4 데이터 저장 및 무결성 보호

**민감 데이터 = 로컬 Only:**
- 거래 내역, 금액, 계좌 정보, 영수증 이미지 등
- File System Access API → `.vault` (원본) + `.bak` (백업) 자동 생성
- IndexedDB = 브라우저 세션 캐시 (Eviction 대비 보조)

**비민감 데이터 = 서버 저장:**
- 유저 계정, 크레딧 잔액, 분석 모드 설정, 초대 코드 등
- PostgreSQL

### 4.5 단일 기기 원칙 × 멀티 디바이스 로그인 UX

> Supabase Auth(이메일 인증)를 사용하기 때문에, 유저가 다른 기기에서 동일 이메일로 로그인할 수 있음.
> 하지만 핵심 재무 데이터는 로컬(.vault)에만 있으므로 새 기기에서는 빈 대시보드가 표시됨.

**해결책: "새 기기 금고 연동" Empty State 전용 화면**

- 다른 기기에서 로그인 시 서버에 로컬 데이터가 없음을 감지
- 전용 안내 화면 표시:
  > "새로운 기기에서 금고를 여셨군요.
  > 금고지기는 완벽한 보안을 위해 기존 기기에만 데이터를 저장합니다.
  > 이전 기기에서 백업한 `.vault` 파일을 여기에 드래그해 금고를 연동하세요."
- `.vault` 파일 드롭존 제공 → 기존 데이터 복원
- 또는 "새 금고로 시작하기" 옵션 제공 (신규 로컬 저장소 생성)

### 4.6 WASM 로컬 OCR 지연 로딩 (Lazy Loading) 전략

> 한국어 특화 OCR(PaddleOCR 등)의 WASM 모듈은 수십 MB로 초기 로딩에 부담.

**구현 방침:**

- 앱 초기 진입 시 OCR 모듈을 로드하지 **않음** (대시보드 렌더링 속도 보장)
- 유저가 최초로 **이미지 파일을 드롭존에 투입하는 시점**에 백그라운드로 WASM 모델 다운로드/초기화
- 로딩 중 UI: 금고 다이얼이 돌아가는 스피너 + "OCR 엔진을 준비하고 있습니다... (최초 1회)" 안내
- 이후 세션에서는 IndexedDB/Cache API에 캐시된 모듈을 즉시 로드 (재다운로드 불필요)
- Phase B(데이터 투입) 구현 시 적용

### 4.7 차트/인사이트 데이터 계약 (Contract) 표준

> AI는 분류/정규화(JSON)까지 담당하고, 차트 렌더링은 프론트엔드가 담당한다.
> 이를 위해 거래 원천 데이터와 집계 모델의 계약을 먼저 고정한다.

**A) 원천 거래 모델 (`Transaction`)**

```ts
type Transaction = {
  id: string
  occurredAt: string // ISO
  amount: number // +수입 / -지출
  direction: 'in' | 'out' | 'transfer'
  merchant: string
  counterparty?: string
  categoryMain: string
  categorySub?: string
  accountType: 'cash' | 'bank' | 'card' | 'loan' | 'investment'
  status: 'confirmed' | 'unconfirmed' | 'pending'
  source: 'csv' | 'xlsx' | 'ocr-local' | 'ocr-premium' | 'manual'
  confidence?: number // 0~1
  isInternalTransfer?: boolean
  tags?: string[]
}
```

**B) 자산 스냅샷 모델 (`WealthSnapshot`)**

```ts
type WealthSnapshot = {
  asOf: string // YYYY-MM-DD
  totalAssets: number
  totalLiabilities: number
  netWorth: number // totalAssets - totalLiabilities
  availableCash: number
  monthlyCashflow: number // 해당 월 수입-지출
  dataConfidence: number // 확정 데이터 비율
}
```

**C) Sankey 뷰 모델 (`SankeyModel`)**

```ts
type SankeyModel = {
  nodes: Array<{ id: string; name: string; group: 'income' | 'pool' | 'expense' | 'debt' | 'saving' }>
  links: Array<{ source: string; target: string; value: number; meta?: { ratio?: number } }>
}
```

### 4.8 실시간 반영 파이프라인 (이벤트 기반)

주기 배치(일/주 단위 새로고침)가 아니라, 데이터 입력 이벤트 기준으로 즉시 갱신:

1. 유저 파일 Drop
2. 파서/OCR 결과를 정규화(`Transaction`)로 변환
3. 로컬 저장 + `vaultStore.transactions` 업데이트
4. selector 재계산(`WealthSnapshot`, `SankeyModel`)
5. 차트/카드/UI 리렌더

**성능 목표 (MVP):**
- CSV/XLSX: 300ms 이내 1차 반영
- 로컬 OCR: 1~3초(최초 WASM 로딩 제외)
- 차트 갱신: 상태 반영 후 즉시(1프레임~수백ms)

### 4.9 AI 챗봇 비용 최적화: 3단계 하이브리드 전략

> 모든 프롬프트를 고성능/고비용 LLM으로 보내지 않는다.
> 유저 입력을 "가벼운 요청"과 "무거운 요청"으로 분기해 비용/속도를 동시에 최적화한다.

#### 1단계: 로컬 룰·경량 분류기 (비용 0원 / 반응 0.1초급)

- **역할:** Intent Routing, 단순 화면 이동/명령 실행
- **예시:** "이번 달 외식비 찾아줘", "여행 예산 페이지 열어줘"
- **기술:** Regex + 키워드 룰 또는 초경량 텍스트 분류 모델(브라우저 실행 가능)
- **동작:** 답변 생성 대신 즉시 라우팅/상태 변경 (`Zustand`) 수행
- **원칙:** 이 단계는 서버 호출 없이 프론트엔드 액션으로 끝낸다

#### 2단계: 저비용 경량 API (건당 저비용 / 반응 1초급)

- **역할:** 데이터 분류, 파싱, 구조화(JSON 추출)
- **예시:** "방금 카카오페이 5만 원은 축의금이야"
- **기술:** 경량 상용 모델(예: GPT-4o-mini/Haiku 급) 또는 동급 저비용 추론 엔진
- **동작:** 자연어를 구조화 JSON으로 변환해 원장/상태를 업데이트
- **출력 예시:**
  ```json
  { "amount": 50000, "category": "축의금" }
  ```

#### 3단계: 고성능 LLM API (건당 고비용 / 반응 2~3초급)

- **역할:** 심층 재무 상담, 시나리오 분석, 생성형 조언
- **예시:** "요즘 식비가 너무 늘었는데 어떻게 예산을 재조정할까?"
- **기술:** 고성능 모델 (GPT-4o / Claude Sonnet 급)
- **방어책:** 무거운 질의는 크레딧 차감 정책과 연결해 비용 상한을 관리

#### 구현 원칙 (Bes2의 늪 회피)

- **프론트엔드에서 무거운 LLM 연산 금지:** 브라우저 과열/프리즈를 피하기 위해 대형 추론은 서버에서 처리
- **행동(Action)은 로컬, 대화(Chat)는 서버:**  
  라우팅/패널 토글/차트 오픈은 로컬 상태 변경으로 처리하고, 서버는 텍스트 응답/분류 JSON만 반환
- **기본 뼈대:** 저비용 경량 API + 프론트엔드 제어 로직(Zustand) 결합

#### 향후 확장 (메뉴 딥다이브 준비)

- 메뉴별 Intent 사전(지기/내역/황금자산/예산&목표/비밀금고) 구축
- 챗 입력창 단일 진입점에서 라우팅 + 데이터 액션 + 상담 호출까지 통합

### 5.1 가치 기반 크레딧 과금 시스템

**이중 과금 구조:**

| 기능 | 과금 방식 | 설명 |
|---|---|---|
| 로컬 AI (분류/파싱) | 최소 과금 | 광고 대신 소액 크레딧 차감. 진입 장벽 낮춤 |

- **크레딧 표시**: 0.1 단위 소수점 디지털 표시
- **결제 연동**: 토스(Toss) 간편 결제 기반 1,000원 / 5,000원 / 10,000원 소액 충전
- **가입 시 무료 크레딧**: 모드별 차등 제공

#### 크레딧 차감의 심리학 (가치 치환 UI)

> "이 영수증 50장과 은행 내역을 직접 대조하고 분류했다면 약 45분이 소요되었을 것입니다.
> (작업 완료 / 소모 크레딧: 1.5 C)"

비용 지출을 **'시간의 구매'**로 치환하여 서비스 가치를 지속적으로 체감시킴.

#### 시간 구매 프레이밍 확장 (TODO: Phase D에서 구현)

- **작업 전 예상 견적**: "이 파일을 분석하면 약 0.3C 소모 예상 (예상 절약 시간: 8분)" → 유저 납득 후 실행
- **월간 절약 리포트**: "이번 달 금고지기가 당신 대신 일한 시간: 4시간 32분" → 충전 유도 자연스러운 넛지
- **온보딩 첫 체험**: 가입 시 샘플 CSV 제공 → 무료 크레딧으로 즉시 "자동 분류 체험" → 가치 체감

### 9.1 프론트엔드 코드 분리안 (즉시 적용)

차트 품질/속도/유지보수를 위해 `store`와 `집계(selector)`를 분리:

```txt
src/
  stores/
    vaultStore.js                # 원천 상태(raw transaction, chat, ui)
  selectors/
    vaultSelectors.js            # 집계/파생 데이터 (snapshot, sankey model, confidence)
  components/
    charts/
      VaultSankeyCard.jsx        # 차트 렌더 계층 (라이브러리 어댑터)
```

**규칙:**
- `store`는 raw data 관리만 담당
- 계산/집계는 selector 함수에서만 수행
- 차트 컴포넌트는 selector 결과를 입력받아 렌더만 수행
